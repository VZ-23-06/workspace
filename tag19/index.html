GIT VERTIEFUNG

  1. Erstelle ein Projekt & eine Repo
  2. Lade das Team ein
  3. Nun erstellst du eine Branch
    • git branch "name"
    • git checkout "name"
    • git checkout -b "name"
    • git branch -d "name"
    • git branch -D "name"
    • git branch

Commits

  • git log --all
  • git log --oneline
  • git log --graph
  • mit "q" kommst du aus der Ansicht wieder raus

Das Mergen

Sobald sämtliche Teams ihre Änderungen erfolgreich in den main Branch eingebracht haben und dieser auf dem neuesten Stand ist, ist es nun an der Zeit, die Inhalte des Main Branches in unseren eigenen Zweig zu mergen

  1. Falls du dich nicht in deiner Branch befindest -> git ...
  2. git merge main -> main wird mit der branch gemerged
  3. git checkout main
  4. lösche die Branch, wenn du sie nicht mehr benötigst ->
    5. git

Commit zurück gehen und Änderungen übernehmen

  1. du kannst git ... verwenden, um die Liste der Commit-IDs anzuzeigen
    • die id auswählen, die man haben möchte -> Vorraussätzung ist, dass du regelmäßig pusht

  2. Du möchtest aus einem bestimmten Commit einen neuen Branch erstellen und auf diesen wechseln.
    • git checkout -b "neuer-branch-name" "commit-id"

  3. git add .
  4. git commit -m "back"
  5. git push origin "neuer-branch-name"

GIT WORKFLOW Variante 1

  • Entwickler arbeiten in ihren Feature-Zweigen und führen regelmäßige Commits durch.
  • Diese Commits werden periodisch in das Remote-Repository gepusht, um die Änderungen zu sichern und die Zusammenarbeit zu erleichtern.
  • Wenn die Feature-Entwicklung fertiggestellt ist und Tests bestanden hat, wird ein Pull Request (oder Merge Request) erstellt.
  • Der Pull Request wird von anderen Teammitgliedern überprüft. Nach der Genehmigung wird der Feature-Zweig in den Hauptzweig gemergt.
  • Der Feature-Zweig wird nur dann gelöscht, wenn seine Änderungen erfolgreich in den Hauptzweig integriert wurden und er nicht mehr benötigt wird.
